# Exact Runtime Agent ID Reuse Design

## Problem

Real test4 proved role-agent routing currently records a generated planned id such as `logos-1-...`, while Claude Code Agent creation returns a different actual runtime `agentId` such as `a403c...`. The next reuse dispatch correctly requires exact resume-by-agent-id, but it points at the generated id, not the runtime id that Claude Code can resume.

Fresh Agent creation on reuse is forbidden because it hides loss of role-agent continuity. Replacement must stay explicit.

## Goals

- Reuse every role agent by the exact runtime Agent tool id returned by first creation.
- Preserve durable role/generation audit identity for `muse`, `logos`, `worker`, `watcher`, and `helper`.
- Keep `state.json` as workflow authority.
- Keep `agent_sessions.json` as routing metadata only.
- Fail closed when reuse is requested before runtime id capture.
- Avoid fresh-agent fallback except explicit replacement.

## Non-goals

- Do not add workflow authority to `agent_sessions.json`.
- Do not depend on an unavailable `SendMessage` tool name.
- Do not silently replace or recreate role agents on reuse failure.
- Do not make planning roles special; all roles share one contract.

## Design

Each role session stores two identities:

- `session_id`: durable logical role/generation id generated by the workflow, for audit and replacement lineage.
- `runtime_agent_id`: actual Claude Code Agent tool id returned after first create dispatch, for exact resume routing.

Existing `agent_id` semantics move to `session_id`. Runtime routing uses `runtime_agent_id` once captured.

### Create flow

1. `resolve_role_session()` creates or reuses a role session.
2. New generation gets `session_id`, status `active`, and empty `runtime_agent_id`.
3. Create dispatch includes `agent_runtime.action = "create"`, plus `session_id`.
4. Main session calls Agent tool.
5. Main session records returned `agentId` through a workflow-owned recorder, e.g. `record_runtime_agent_id(run_root, role, session_id, runtime_agent_id)`.
6. Registry persists `runtime_agent_id` and appends role notebook entry.

### Reuse flow

1. `resolve_role_session()` sees active role session.
2. If `runtime_agent_id` exists, runtime contract is:
   - `action = "resume_by_agent_id"`
   - `agent_id = runtime_agent_id`
   - `session_id = session_id`
   - `must_resume_exact_agent_id = true`
3. If `runtime_agent_id` is missing, the workflow fails closed with a clear error. The error states that role creation happened but actual runtime Agent id was never recorded.

### Replacement flow

Explicit replacement increments generation, writes a new `session_id`, clears `runtime_agent_id`, and emits create routing. The replacement path does not reuse the previous runtime id.

## Data shape

`agent_sessions.json` role entry:

```json
{
  "session_id": "logos-1-...",
  "runtime_agent_id": "a403c54784811b8a7",
  "generation": 1,
  "status": "active",
  "created_at": "...",
  "last_seen_at": "...",
  "log_path": "agents/logos-log.md",
  "summary_path": "agents/logos-summary.md",
  "last_dispatch_owner": "grill-storm-loop",
  "replacement_reason": ""
}
```

For compatibility during migration, loader may accept old `agent_id` as `session_id` when `session_id` is absent. It must not treat old `agent_id` as `runtime_agent_id`.

## Error handling

Missing `runtime_agent_id` on reuse is a hard blocker. It should not call Agent, not create a fresh role agent, and not mark replacement. The operator must either record the missing runtime id from creation evidence or use explicit replacement flow.

## Tests

- `resolve_role_session()` first create returns `session_id`, empty `runtime_agent_id`, and runtime action `create`.
- `record_runtime_agent_id()` persists actual Agent tool id and rejects wrong role/session pairing.
- Second `resolve_role_session()` for same role returns reuse contract using `runtime_agent_id` as `agent_runtime.agent_id`.
- Active reuse without `runtime_agent_id` fails closed.
- `replace_role_session()` increments generation, changes `session_id`, clears `runtime_agent_id`, and emits create routing.
- Grill-storm Muse/Logos dispatches show both ids and reuse actual runtime id.
- Orchestrator worker/watcher/helper dispatches use same runtime contract.
- Docs mention session id vs runtime agent id and forbid fresh-agent fallback.

## Acceptance

A real bench run must stop using generated role ids for runtime resume. After first role creation, subsequent dispatches must route to the actual Agent tool `agentId` captured from creation.
